1.1 LONG SHORT-TERM MEMORY (LSTM) - DEEP EXPLANATION
---

WHAT IS LSTM?
LSTM is a type of Recurrent Neural Network (RNN) that can learn long-term dependencies
in sequential data. Unlike traditional RNNs that forget old information, LSTMs have a
"memory cell" that can retain information over many timesteps.

WHY LSTM FOR SMART GRID?
- Power grid data is sequential (time-series data)
- Attacks unfold over multiple timesteps (slow drift, sustained patterns)
- Simple neural networks can't capture this temporal context
- LSTM learns both immediate anomalies AND patterns that develop over time

LSTM ARCHITECTURE (YOUR PROJECT):
```
Input Layer:
├─ Physical: [voltage, current, frequency, power, response_time] (5 features)
├─ Cyber: [latency, packet_loss, integrity, comm_freq] (4 features)
│
├─ Sequence: 24 timesteps of each feature
└─ Total input per agent: 24 × 9 = 216 values per sequence

LSTM Layer (Physical Branch):
├─ Input: 24 timesteps × 5 features
├─ LSTM cells: 64 hidden units
├─ Process: Each timestep updates hidden state h_t and cell state c_t
├─ Output: 64-dimensional context vector
└─ Learns patterns like: voltage trending up = trouble

LSTM Layer (Cyber Branch):
├─ Input: 24 timesteps × 4 features
├─ LSTM cells: 64 hidden units
├─ Output: 64-dimensional context vector
└─ Learns patterns like: packet loss increasing = network problem

Merge & Dense Layers:
├─ Concatenate: [physical_64d, cyber_64d] = 128d vector
├─ Dense(32): Learn interaction between physical+cyber
├─ Dense(1) + Sigmoid: Output probability P(anomaly)
│
└─ Final output: Single value between 0.0 and 1.0
   - 0.0 = definitely normal
   - 1.0 = definitely anomalous
   - 0.5 = uncertain
```

HOW LSTM CELL WORKS (Internal Mechanism):
```
At each timestep t:

1. FORGET GATE (what to discard from memory?)
   f_t = sigmoid(W_f × [h_{t-1}, x_t] + b_f)
   Decides: "Should I forget old information?"
   - Output: 0 = forget, 1 = keep

2. INPUT GATE (what new info to add?)
   i_t = sigmoid(W_i × [h_{t-1}, x_t] + b_i)
   C̃_t = tanh(W_c × [h_{t-1}, x_t] + b_c)
   Decides: "What new information should I store?"
   - i_t = gate (0=don't add, 1=add)
   - C̃_t = candidate values

3. CELL STATE UPDATE (update memory)
   C_t = f_t ⊙ C_{t-1} + i_t ⊙ C̃_t
   Combines: (forget × old_memory) + (input_gate × new_info)
   This is the long-term memory

4. OUTPUT GATE (what to output?)
   o_t = sigmoid(W_o × [h_{t-1}, x_t] + b_o)
   h_t = o_t ⊙ tanh(C_t)
   Decides: "What relevant information should I pass forward?"
   - o_t = gate (0=hide, 1=reveal)
   - h_t = current hidden state (short-term memory)
```

EXAMPLE: LSTM Learning FDI Attack Pattern
```
Normal operation (t=1 to t=20):
└─ Voltage: [230, 230, 230, 230, ...] (stable)
└─ Current: [15, 15, 15, 15, ...] (stable)
└─ LSTM hidden state: s_20 = "grid is stable"
└─ Output probability: P(anomaly) = 0.05

Attack starts (t=21 to t=30):
└─ Voltage: [230, 235, 240, 245, 250, 255, 260, 265, 270, 275] (increasing bias)
└─ Current: [15, 15, 15, 15, 15, 15, 15, 15, 15, 15] (normal)
└─ LSTM forgets old "normal" state gradually (forget gate activates)
└─ LSTM inputs new pattern: "voltage increasing without current increase"
└─ At t=25: P(anomaly) = 0.72 (LSTM detected drift pattern)
└─ At t=30: P(anomaly) = 0.95 (very confident attack)

Simple deviation scoring misses this:
└─ At t=21: |235-230|/threshold = small, flags as normal
└─ At t=25: |250-230|/threshold = moderate, maybe flags
└─ But LSTM sees the TREND and flags earlier
```

WHY YOUR PROJECT USES DUAL-BRANCH LSTM:
```
Physical and Cyber Features are different:

PHYSICAL FEATURES:                  CYBER FEATURES:
- Change slowly (equipment inertia) - Change rapidly (network response)
- Small variations = normal          - Large variations = normal
- Physical laws apply               - No physical constraints
- Baseline: 230V is normal          - Baseline: 20ms latency is normal
  for generator                        for network

Solution: Separate LSTM branches
├─ Physical branch learns slow patterns (gradual voltage decline)
├─ Cyber branch learns fast patterns (sudden latency spike)
├─ Then merge: if BOTH branches flag = high confidence attack
├─ If only physical: might be equipment fault
└─ If only cyber: might be network glitch
```

TRAINING YOUR LSTM:
```
1. Data Preparation:
   - 24-hour simulation: 288 timesteps × 100 agents = 28,800 timesteps
   - Create 24-timestep windows: 28,776 sequences
   - Split: 80% train (23,020), 20% validation (5,756)
   - Class imbalance: 99% normal, 1% anomalous
   - Solution: Oversample attacks (1.35×) + Focal Loss

2. Loss Function (Focal Loss):
   FL(p_t) = -α(1-p_t)^γ log(p_t)
   - α = 0.65 (weight for minority class)
   - γ = 2.0 (focus on hard examples)
   - Why? Normal examples are easy to predict
     But attacks are hard, so focus training on attacks

3. Optimizer:
   - Adam (adaptive learning rate)
   - Learning rate: 0.001 initially, decay to 0.0001
   - Batch size: 32
   - Epochs: 50-100

4. Calibration (Temperature Scaling):
   - Raw LSTM output: P_raw = 0.8 (overconfident)
   - Apply temperature: P_calibrated = softmax(logits / T)
   - T = 1.5 (found on validation set)
   - Result: P_calibrated = 0.65 (more realistic probability)
   - Why? So P(anomaly)=0.65 actually means 65% confidence
```

LSTM IN YOUR DETECTION PIPELINE:
```
Detection Flow:
1. Observation arrives (voltage, current, latency, ...)
2. History window filled (last 24 timesteps)
3. Physical branch LSTM: P_phys = predict probability
4. Cyber branch LSTM: P_cyber = predict probability
5. Fusion: P_fused = 0.5 × P_phys + 0.5 × P_cyber
6. Combine with deviation score and behavioral signatures
7. Multi-layer detection layers (CUSUM, DoS, MITM)
8. Final decision: Flag if any layer triggers
```

1.2 REINFORCEMENT LEARNING - FUNDAMENTALS
---

WHAT IS REINFORCEMENT LEARNING (RL)?
RL is a machine learning paradigm where an agent learns by interacting with an
environment, taking actions, and receiving rewards/penalties.

KEY CONCEPTS:
```
Agent: Decision maker (our audit scheduler)
Environment: The smart grid (100 agents, their risk states)
State: Current situation (how risky is each agent?)
Action: What to do (increase/decrease/hold audit frequency)
Reward: Feedback (did we catch the attack? Did we waste resources?)
Trajectory: Sequence of (state, action, reward) over time
```

RL vs SUPERVISED LEARNING:
```
Supervised Learning:
├─ Input: X (features)
├─ Output: Y (labels)
├─ Training: Compare prediction to ground truth
└─ Example: LSTM classification ("is this anomalous?")

Reinforcement Learning:
├─ Input: State S (current risk scores)
├─ Output: Action A (audit frequency)
├─ Training: No ground truth! Only reward signal
├─ Agent must learn through trial & error
└─ Example: Q-learning scheduler ("how often should we audit?")
```

WHY RL FOR AUDIT SCHEDULING?
```
Problem: We don't know the "optimal" audit frequency a priori
- If we audit too often: waste resources (cost ↑)
- If we audit too rarely: miss attacks (risk ↑)
- The answer depends on real-time risk, which changes

Solution: Use RL to learn the optimal trade-off
- Start with random auditing
- Observe outcomes (did we catch attacks? Did we waste money?)
- Gradually improve scheduling decisions
- Converge to optimal policy
```

1.3 Q-LEARNING - DETAILED ALGORITHM
---

WHAT IS Q-LEARNING?
Q-learning is a model-free reinforcement learning algorithm that learns the value
of taking each action in each state. Q = "quality of action"

Q-VALUE DEFINITION:
Q(s,a) = expected discounted future reward when taking action a in state s

EXAMPLE:
```
State: Agent 5 risk is HIGH, hasn't been audited recently
Action options:
├─ INCREASE_AUDIT: Audit this agent more frequently
│  └─ Q(HIGH, INCREASE) = +2.5 (catches more attacks)
├─ HOLD: Keep current frequency
│  └─ Q(HIGH, HOLD) = -0.5 (might miss attack)
└─ DECREASE_AUDIT: Audit less
   └─ Q(HIGH, DECREASE) = -5.0 (very risky)

Best action: INCREASE_AUDIT (highest Q-value)
```

Q-LEARNING UPDATE RULE (Bellman Equation):
```
Q(s,a) ← Q(s,a) + α[r + γ max_a' Q(s',a') - Q(s,a)]

Where:
├─ Q(s,a) = old estimate
├─ α = learning rate (0.01 to 0.1) = how much to update
├─ r = immediate reward from action
├─ γ = discount factor (0.9) = importance of future vs immediate
├─ max_a' Q(s',a') = best future value
└─ [r + γ max_a' Q(s',a') - Q(s,a)] = TD error (prediction error)

Intuition:
- If actual outcome is better than predicted: increase Q
- If actual outcome is worse than predicted: decrease Q
- Converges to true Q-values over time
```

Q-TABLE FOR YOUR PROJECT:
```
State Space:
├─ Anomaly Score: Low (0-1), Medium (1-2), High (2+)
├─ Risk Score: Low, Medium, High
└─ Combinations: 3 × 3 = 9 states per agent

Action Space:
└─ INCREASE_AUDIT, HOLD, DECREASE_AUDIT (3 actions)

Q-Table size per agent:
└─ 9 states × 3 actions = 27 Q-values

Total Q-table size for 100 agents:
└─ 100 × 27 = 2,700 values (tiny! why this is good)

Example Q-Table for Agent SUB-05:
```
          INCREASE    HOLD   DECREASE
LOW         0.2      0.5      -0.3
MEDIUM      1.5      0.1      -1.2
HIGH        2.8      0.4      -5.0
```

When state=(HIGH risk, HIGH anomaly):
- Best action = INCREASE_AUDIT (Q=2.8)
- This makes sense: high risk → audit more
```

Q-LEARNING TRAINING IN YOUR PROJECT:
```
Episode = 288 timesteps (one 24-hour cycle)

For each timestep t:
  1. Observe current state s_t (anomaly scores, risk of all agents)
  2. Choose action a_t using ε-greedy:
     ├─ With probability ε (0.3 → 0.05): random action (explore)
     └─ With probability 1-ε: best action (exploit)

  3. Environment responds with:
     ├─ New state s_{t+1}
     ├─ Reward r_t (did we catch attacks? waste resources?)
     └─ Continue? (True if t < 288)

  4. Update Q-value:
     Q(s_t, a_t) ← Q(s_t, a_t) + α[r_t + γ max_a Q(s_{t+1}, a) - Q(s_t, a_t)]

  5. Move to next state: s_t ← s_{t+1}

Repeat for multiple episodes (200 episodes typical)
After ~100 episodes, Q-values converge to optimal policy
```

REWARD FUNCTION (YOUR PROJECT):
```
Base reward: r = 0 (no attack, no audit = baseline)

Add penalties/bonuses:
├─ Missed attack: r -= 11.0
│  └─ Very negative! Missing attacks is worst outcome
│
├─ Audit action: r -= 0.03
│  └─ Small cost per audit (11/0.03 = 367:1 ratio)
│  └─ System is very attack-averse (prioritizes security)
│
├─ Proactive audit on high-risk: r += 2.5
│  └─ Bonus for auditing before attacks escalate
│
└─ Under-auditing high-risk: r -= 5.0 (quadratic penalty)
   └─ Penalty if high-risk agents aren't audited enough

Example Episode:
Time t1: Low-risk agent, do nothing
  └─ r = 0 (no action taken)

Time t5: High-risk agent appears, audit it immediately
  └─ r = +2.5 (proactive bonus) - 0.03 (audit cost) = +2.47

Time t10: Attack happens on audited agent
  └─ Caught! r = 0 (normal)

Time t15: Attack happens on un-audited agent
  └─ Missed! r = -11.0 (severe penalty)
  └─ Q-value for that action decreases dramatically
  └─ Next time, Q-learning will audit more

After 200 episodes:
Q-learning learns: "Don't be cheap with audits! Security > cost"
```

ε-GREEDY EXPLORATION:
```
Exploration-Exploitation Dilemma:
├─ Exploitation: Use best known action (high Q-value)
├─ Exploration: Try random action (might find better)

Solution: ε-Greedy
├─ With prob ε: Random action (explore)
├─ With prob 1-ε: Best action (exploit)

Your project:
├─ Initial ε = 0.3 (30% random, 70% best known)
├─ Decay over episodes: ε ← ε × 0.995
├─ Final ε = 0.05 (5% random, 95% best known)
├─ Reason: Early training needs exploration
│  Later training should exploit what we learned
```

1.4 HYBRID SCHEDULING (Q-LEARNING + GRADIENT DESCENT)
---

THE PROBLEM WITH Q-LEARNING ALONE:
```
Q-learning output:
├─ Discrete actions: INCREASE, HOLD, DECREASE
├─ No continuous refinement
├─ Jumps between frequency f=1, f=2, f=3

Real problem:
├─ Optimal audit frequency might be f=2.3
├─ Q-learning can only do f=2 or f=3
├─ Wastes resources with discrete jumps
```

THE PROBLEM WITH GRADIENT DESCENT ALONE:
```
Gradient descent alone:
├─ Knows cost function: C = C_a × f + C_f × (R/f)
├─ Can find optimal f* exactly
├─ But doesn't know which direction to go first!
├─ Might optimize the wrong thing
```

THE HYBRID SOLUTION:
```
Step 1: Q-Learning (Decide Direction)
├─ Input: Current risk state
├─ Output: Direction (INCREASE, HOLD, DECREASE)
├─ This answers: "Should we audit MORE or LESS?"

Step 2: Gradient Descent (Decide Magnitude)
├─ Input: Current frequency f_i, risk R_i
├─ Cost function: C(f) = C_a × f + C_f × (R/f)
├─ dC/df = C_a - C_f × (R / f²)
├─ Gradient descent: f_new = f - lr × dC/df
├─ Output: Optimal f*
└─ This answers: "By how much should we change the frequency?"

Analogy:
├─ Q-learning: GPS compass says "go NORTHEAST"
├─ Gradient descent: Calculates exact distance to destination
└─ Together: Navigate optimally AND efficiently
```

COST FUNCTION EXPLAINED:
```
C(f) = C_a × f + C_f × (R / f)

Two competing terms:

Term 1: C_a × f (Audit Cost)
├─ Increases with frequency f
├─ More audits = higher cost
├─ C_a = $100 per audit per 24h
├─ If f=1: cost = $100
├─ If f=5: cost = $500

Term 2: C_f × (R / f) (Failure Cost)
├─ Decreases with frequency f
├─ More audits = catch more attacks = lower failure cost
├─ C_f = $1000 (cost of undetected attack)
├─ R = current risk score (0-10)
├─ If f=1: cost = $1000 × 10 / 1 = $10,000 (very risky)
├─ If f=5: cost = $1000 × 10 / 5 = $2,000 (safer)

Total Cost: C = audit_cost + failure_cost
├─ Too low f: failure cost dominates (attacks slip through)
├─ Too high f: audit cost dominates (waste resources)
├─ Optimal f*: Minimizes total cost
```

FINDING OPTIMAL FREQUENCY:
```
Analytical Solution (takes derivative):
├─ dC/df = C_a - C_f × (R / f²)
├─ Set dC/df = 0:
│  ├─ C_a = C_f × (R / f²)
│  ├─ f² = C_f × R / C_a
│  └─ f* = √(C_f × R / C_a)
│
├─ Example: C_a=100, C_f=1000, R=5
│  └─ f* = √(1000 × 5 / 100) = √50 = 7.07 audits/24h
│
└─ Intuition: Higher risk R → more audits; higher cost C_a → fewer audits

Gradient Descent Solution (iterative):
├─ Start with current f_i
├─ Compute gradient: g = dC/df = C_a - C_f × (R / f²)
├─ Update: f_new = f - learning_rate × g
├─ Repeat until convergence
├─ Learning rate: 0.01 (small steps)
└─ Converges to f* in 5-20 iterations
```

YOUR HYBRID SCHEDULER FLOW:
```
For each agent i at timestep t:

1. Q-LEARNING DECISION:
   ├─ State: (anomaly_bin, risk_bin)
   ├─ Action: Q-table lookup → best_action
   │  └─ action ∈ {INCREASE, HOLD, DECREASE}
   ├─ Decide direction:
   │  ├─ INCREASE: f_new = f + 0.5
   │  ├─ HOLD: f_new = f
   │  └─ DECREASE: f_new = f - 0.5
   └─ Get temporary frequency: f_temp

2. GRADIENT DESCENT REFINEMENT:
   ├─ Cost function: C(f) = 100f + 1000(R_i/f)
   ├─ Gradient: dC/df = 100 - 1000×R_i / f²
   ├─ Update: f ← f - 0.01 × gradient
   ├─ Repeat 10 times (convergence)
   └─ Get refined frequency: f_refined

3. APPLY CONSTRAINTS:
   ├─ Clamp: f_refined = max(0.5, min(5, f_refined))
   │  └─ f must be between 0.5 and 5 audits per 24h
   ├─ Global budget:
   │  ├─ Sum all f_i
   │  ├─ If total > budget: scale all f_i down
   │  └─ Ensure sum(f_i) ≤ B (budget)
   └─ Final: f_i ← f_refined

4. RESULT:
   └─ Agent i will be audited f_i times in the next 24h cycle
```

EXAMPLE WALKTHROUGH:
```
Agent SUB-05:
├─ Current risk R_i = 6.5 (high)
├─ Current frequency f = 2.0 audits/24h
├─ Base audit cost C_a = 100
├─ Failure cost C_f = 1000

Step 1: Q-Learning
├─ Anomaly score: 2.8 (HIGH)
├─ Risk: 6.5 (HIGH)
├─ State: (HIGH, HIGH)
├─ Q(HIGH, INCREASE) = 2.8 (best action)
├─ Q(HIGH, HOLD) = 0.4
├─ Q(HIGH, DECREASE) = -5.0
├─ Decision: INCREASE
├─ Temporary: f_temp = 2.0 + 0.5 = 2.5

Step 2: Gradient Descent
├─ Cost at f=2.0: C = 100×2.0 + 1000×6.5/2.0 = 200 + 3250 = 3450
├─ Cost at f=2.5: C = 100×2.5 + 1000×6.5/2.5 = 250 + 2600 = 2850
├─ Cost at f=3.0: C = 100×3.0 + 1000×6.5/3.0 = 300 + 2167 = 2467
├─ Gradient at f=2.5: dC/df = 100 - 1000×6.5/(2.5²) = 100 - 1040 = -940
│  └─ Negative gradient means cost decreases as f increases
│  └─ Keep increasing f
├─ Update: f = 2.5 - 0.01×(-940) = 2.5 + 9.4 = 11.9
│  └─ Too big jump! Limit by constraints
└─ Refine with several iterations, converge to optimal f* ≈ 8.06

Step 3: Constraints
├─ f_refined = 8.06, but max allowed = 5
├─ Clamp: f_refined = min(5, 8.06) = 5.0
├─ Check global budget: If sum(all f_i) > B
│  └─ Scale all proportionally
└─ Final: f_i = 5.0 audits per 24h (maximum allowed)

Result: SUB-05 will be audited 5 times in next 24h
├─ Q-learning knew: increase frequency (risky agent)
├─ Gradient showed: optimal f* would be 8.06
├─ But constraints limited it to max=5
└─ System is consistent and explainable
```

WHY HYBRID IS BETTER THAN EITHER ALONE:
```
Q-Learning Only:
├─ Would output: INCREASE
├─ Adjust frequency by fixed +0.5
├─ Result: f = 2.5 (discrete jump)
├─ Misses optimal value 8.06

Gradient Descent Only:
├─ Would optimize f starting from f=2.0
├─ But doesn't know risk has increased!
├─ Might refine to some local minimum
├─ Lacks direction signal from RL

Hybrid:
├─ Q-learning provides direction: "go up"
├─ Gradient finds how much: "go up by 6.06"
├─ Together: optimal, informed, precise
```

Q-Learning Fundamentals:
├─ Learns action values: Q(s,a) = expected discounted future reward
├─ Update rule (Bellman): Q(s,a) ← Q(s,a) + α[r + γ max Q(s',a') - Q(s,a)]
├─ Converges to optimal policy after ~100 episodes
├─ No model needed (model-free RL)

Your Application:
├─ State: (anomaly_score, risk_score) for each agent
├─ Actions: INCREASE_AUDIT, HOLD, DECREASE_AUDIT

Why Q-Learning:
├─ Offline training possible (learn from simulation)
├─ Small state space (27 states = 27 Q-values per agent)
├─ Converges fast (200 episodes = convergence)
├─ Interpretable (inspect Q-table to see learned policy)
└─ Better than deep RL for small problems


Q8: Explain the hybrid Q-Learning + Gradient Descent scheduler.

A8: [See Section 1.4 - Hybrid Scheduling Detailed Explanation above]

Problem It Solves:
├─ Q-learning alone: Discrete actions (f=1, f=2, f=3)
├─ Misses optimal continuous values (e.g., f=2.37)
├─ Gradient descent alone: Doesn't know direction
└─ Need both direction (from RL) and magnitude (from calculus)

Your Solution (Two-Phase):
Phase 1 - Q-Learning (Direction):
├─ Input: Current state (risk, anomaly)
├─ Lookup: Q(s,a) for each action
├─ Choose: Best action (INCREASE, HOLD, DECREASE)
├─ Output: Direction (which way to move f)

Phase 2 - Gradient Descent (Magnitude):
├─ Cost function: C(f) = C_a×f + C_f×(R/f)
├─ Gradient: dC/df = C_a - C_f×(R/f²)
├─ Update: f_new = f - learning_rate × gradient
├─ Repeat: Until convergence (10 iterations typical)
└─ Output: Optimal frequency f*

Result:
├─ Q-learning picks direction: "audit more"
├─ Gradient finds optimal value: "audit 3.7 times per 24h"
├─ Together: Better than either alone

Example:
├─ Agent with risk R=8.5, current f=2.0
├─ Q-learning says: INCREASE (Q>0)
├─ Gradient calculates: f* = √(1000×8.5/100) = 9.2
├─ Constraints limit: f = min(9.2, max_allowed=5) = 5.0
└─ Final: Audit this agent 5 times per 24h (maximum)

Q13: What does "3-modality voting ensemble" mean?

A13: Your system uses 3 independent detection methods that "vote" on whether an alert is real.

The 3 Modalities:

MODALITY 1: Deviation Scoring
├─ What: Compare observed values to baseline
├─ Formula: S = √(mean((x-bx)/thx)²) + √(mean((y-by)/thy)²)
├─ Answers: "Is the current reading far from normal?"
├─ Example: voltage 250V vs baseline 230V → flag
├─ Catches: Immediate, large deviations
└─ Misses: Slow changes (1% per timestep)

MODALITY 2: LSTM (Dual-Branch)
├─ What: Learn temporal patterns in history
├─ Input: 24 timesteps of physical and cyber features
├─ Output: P(anomaly) probability (0-1)
├─ Answers: "Does the recent pattern match attack signatures?"
├─ Example: Voltage trending slowly up → increasingly suspicious
├─ Catches: Temporal patterns, sustained anomalies
└─ Misses: Sudden spikes (too fast for LSTM to react)

MODALITY 3: Behavioral Signatures
├─ What: Detect specific patterns (step, ramp, oscillation)
├─ Formulas:
│  ├─ Step: |current_score - prev_score| > 2σ
│  ├─ Ramp: Linear regression slope > threshold
│  └─ Oscillation: Autocorr peak at 1-10 Hz
├─ Answers: "Do we see domain-specific attack patterns?"
├─ Example: Voltage increasing by exactly 1V per timestep → suspect
├─ Catches: Attack-like temporal signatures
└─ Misses: Novel patterns not in signature database

How Voting Works:

Single Modality Alert:
├─ Deviation high, but LSTM normal, no behavior?
└─ Probably noise (load fluctuation) → SUPPRESS

Double Modality Alert:
├─ Deviation high AND LSTM confirms?
└─ Moderate confidence → INVESTIGATE

Triple Modality Alert:
├─ Deviation high AND LSTM confirms AND behavior present?
└─ High confidence → ESCALATE TO AUDIT

OR-Logic with Suppression:
```
IF (Deviation HIGH) THEN
  IF (LSTM NORMAL) AND (No Behavior) THEN
    SUPPRESS (likely noise)
  ELSE
    FLAG (real anomaly)
ELSE
  NORMAL (no deviation)
```

Example Scenario 1 (False Positive):
├─ Load ramp at substation (normal operation)
│  └─ Causes: Voltage drop 231V → 229V (small)
│
├─ Modality 1 (Deviation): (2/12)² = 0.028 (small)
├─ Modality 2 (LSTM): Sees load pattern from training
│  └─ P(anomaly) = 0.12 (normal)
├─ Modality 3 (Behavior): Ramp present, but known load profile
│  └─ Signature match: "normal load increase"
│
└─ Result: All vote NORMAL → Not flagged ✓ (correct)

Example Scenario 2 (True Positive):
├─ FDI attack on voltage (false data injection)
│  └─ Causes: Voltage jumps 230V → 250V at t=50
│
├─ Modality 1 (Deviation): (20/12)² = 2.78 (high)
├─ Modality 2 (LSTM): Sees sudden jump not in training
│  └─ P(anomaly) = 0.85 (suspicious)
├─ Modality 3 (Behavior): Step change detected (jump pattern)
│  └─ Signature: "FDI spike detected"
│
└─ Result: All vote ANOMALY → Flagged ✓ (correct)

Why 3 Modalities Beat 1:

Single Modality Problems:
├─ Deviation Only: FPR=9%, Recall=37%
├─ LSTM Only: FPR=0%, Recall=23%
└─ Behavior Only: FPR=8%, Recall=45%

Combined (Voting):
├─ FPR=1.81% (lower than any single)
├─ Recall=75.76% (higher than any single)
└─ F1=82.25% (best balance)

Intuition:
├─ Each modality has different false positive triggers
├─ Voting filters out each modality's unique noise
├─ Only flags when multiple agree
└─ Result: High confidence, low false positives

Q16: How do you handle the class imbalance problem (99% normal, 1% attacks)?

A16: Imbalanced data is the biggest problem in anomaly detection.

The Problem:

Training data distribution:
├─ Normal observations: 28,600 samples (99%)
├─ Attack observations: 200 samples (1%)
└─ Ratio: 143:1

What Happens Without Handling:

Standard LSTM training:
├─ Loss function: Cross-entropy
├─ Prediction: model.predict(x)
│  ├─ If normal: return 0.99999 (confident)
│  ├─ If attack: return 0.00001 (confident it's normal!)
│
├─ Why? Model learns:
│  ├─ "Predict everything as normal"
│  └─ "Accuracy will be 99% automatically"
│
└─ Result: LSTM never flags anything (useless)

Your Solutions (3-Pronged Approach):

SOLUTION 1: Oversampling
├─ Problem: 143 normal samples per attack sample
├─ Solution: Replicate attack samples until balanced
├─ Method: Random oversampling (1.35× multiplier)
├─ Result: 28,600 normal, 270 attack (ratio: 106:1)
├─ Trade-off: Slight overfitting, but better than ignoring minority class

SOLUTION 2: Focal Loss
├─ Problem: Model gets overconfident on easy samples
├─ Solution: Weight hard examples more heavily
├─ Formula: FL(p_t) = -α(1-p_t)^γ log(p_t)
│
├─ How it works:
│  ├─ α = 0.65 (weight minority class)
│  ├─ γ = 2.0 (focus exponent)
│  ├─ When p_t = 0.9 (easy, confident prediction):
│  │  └─ Loss = 0.65 × (1-0.9)^2 × log(0.9) ≈ small
│  ├─ When p_t = 0.5 (hard, uncertain prediction):
│  │  └─ Loss = 0.65 × (1-0.5)^2 × log(0.5) ≈ large
│  └─ Result: Model focuses on hard examples (rare attacks)
│
└─ Effect: Prevents model from ignoring minority class

SOLUTION 3: Temperature Scaling (Calibration)
├─ Problem: LSTM outputs overconfident probabilities
├─ Solution: Scale softmax by temperature T
├─ Formula: P_calibrated = softmax(logits / T)
│
├─ Before calibration:
│  ├─ Model says: P(attack) = 0.95 (very confident)
│  ├─ Reality: Only 50% of cases with this pattern are attacks
│  └─ Problem: Overconfident! (miscalibrated)
│
├─ After calibration (T=1.5):
│  ├─ Model says: P(attack) = 0.65 (more realistic)
│  ├─ Reality: 65% of cases with this pattern are attacks
│  └─ Solution: Better aligned! (calibrated)
│
└─ Method: Adjust T on validation set to minimize calibration error

Result:

Training data after handling:
├─ Oversampled: 28,600 normal, 270 attack
├─ Weighted: α=0.65 emphasizes minority
├─ Calibrated: T=1.5 normalizes confidence
└─ Effect: LSTM learns to detect both classes equally

LSTM Performance After Handling Imbalance:
├─ Accuracy: 86.35% (trains without collapsing to always-normal)
├─ Recall: 22.67% (catches some attacks, but not all)
├─ Precision: 100.00% (no false positives)
└─ Issue: LSTM alone is too conservative (needs other layers)

Full System After Handling:
├─ Accuracy: 94.23% (multi-layer catch remaining attacks)
├─ Recall: 75.76% (catches most attacks)
├─ Precision: 89.95% (high confidence)
└─ Solution: Multi-layer ensemble catches what LSTM misses
