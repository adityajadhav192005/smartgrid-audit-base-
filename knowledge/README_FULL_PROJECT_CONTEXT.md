# SmartGrid AI Audit Framework: Full Project Context

## 1. What This Project Is

This project is an end-to-end cyber-physical smart-grid monitoring and audit framework.

It combines:

- a simulated multi-agent smart-grid experiment environment
- a live Rapid SCADA telemetry path
- anomaly detection
- explainable deviation-based risk scoring
- audit decision logic
- response and mitigation views
- a web dashboard with separate experiment and live workspaces

The project is positioned as an operational prototype, not a field-deployed utility system.

Important honesty note:

- the live SCADA pipeline is real
- the live data path from Rapid SCADA to the backend to the dashboard is real
- the current live telemetry values are generated by Rapid SCADA calculated channels
- they are not direct field measurements from physical devices

---

## 2. What It Connects With

The project connects the following layers:

1. **Rapid SCADA**
   - acts as the supervisory telemetry source
   - contains the live 100-agent grid
   - serves current values through its Web API

2. **PowerShell bridge**
   - pulls SCADA data through authenticated API access
   - sends normalized SCADA payloads to the backend
   - supports batched ingest to avoid rate-limit instability

3. **FastAPI backend**
   - receives SCADA payloads
   - converts tags to cyber-physical feature vectors
   - runs the merged live experiment pipeline with deviation scoring, optional LSTM probability, behavior adaptation, hybrid audit scheduling, response, and XAI output
   - exposes API endpoints for the web dashboard and operators

4. **Experiment runner**
   - simulates attacks, faults, agent behavior, and audit scheduling
   - runs RL, gradient-based refinement, hybrid audit allocation, and evaluation
   - exports logs and summary metrics

5. **Next.js dashboard**
   - shows experiment analytics in one workspace
   - shows Rapid SCADA live monitoring in another workspace
   - exposes audit trail, response workflow, XAI, risk analytics, topology, and system health

---

## 3. Two Main Operational Modules

### 3.1 Experiment Running

This is the simulation and benchmarking module.

It is used to:

- create agent populations at different sizes such as `N=100`, `N=200`, and `N=500`
- inject cyber attacks and synthetic faults
- run anomaly detection and audit scheduling over a full simulation horizon
- compare the dynamic strategy against a fixed baseline strategy
- compute metrics such as cost efficiency, risk mitigation, attack-rate reduction, precision, recall, F1, and accuracy

### 3.2 Rapid SCADA Live

This is the real-time operational monitoring module.

It is used to:

- read live Rapid SCADA telemetry from the 100-agent grid
- convert SCADA values to physical and cyber feature vectors
- run the merged live experiment pipeline on each batch
- compute anomaly score, anomaly probability, risk, audit action, response, and explanation
- display live agent state, alerts, audit actions, explanations, and pipeline health in the dashboard

### 3.3 Why The Two Workspaces Exist

The dashboard is intentionally split into:

- `Experiment Running`
- `Rapid SCADA Live`

This separation prevents experimental outputs from mixing with operational monitoring outputs.

## 4. Live SCADA Topology

The validated live Rapid SCADA deployment uses a fixed 100-agent grid:

- `GEN-01` to `GEN-20` : generators
- `SUB-21` to `SUB-50` : substations
- `PMU-51` to `PMU-75` : PMUs
- `BRK-76` to `BRK-100` : breakers

The live model now has two SCADA layers:

- a `300`-channel base calculated setup in `rapidscada_demo/ProjectBaseXML/Cnl.xml`
- a `370`-channel cyber addon in `rapidscada_demo/ProjectBaseXML/Cnl_Cyber_Addon.xml`

Together, the enriched live SCADA model exposes `670` configured channels.

The addon supplies the missing live cyber telemetry slots expected by the backend:

- generators: `latency`, `packet_loss`, `integrity`, `comm_freq`
- substations: `packet_loss`, `integrity`, `comm_freq` while existing substation latency remains in the base model
- PMUs: `latency`, `packet_loss`, `integrity`, `comm_freq`
- breakers: `latency`, `packet_loss`, `integrity`, `comm_freq`

## 5. End-to-End System Flow

### 5.1 Live SCADA Flow

1. Rapid SCADA generates current values for the 100-agent grid through calculated channels.
2. The PowerShell bridge reads those values from the Rapid SCADA Web API.
3. The backend normalizes raw SCADA tags into physical and cyber feature vectors.
4. The live experiment pipeline optionally scores the last `24` windows with the live LSTM checkpoint.
5. Deviation scoring computes physical and cyber deviation against baselines and thresholds.
6. Behavior adaptation and clustering update the live context.
7. The hybrid scheduler computes audit frequency and current audit action.
8. Response logic assigns mitigation state and downstream risk.
9. Feature-level explanation is generated.
10. The dashboard displays live score, anomaly probability, risk, audit status, response, and explanation from the fresh live snapshot only.

### 5.2 Experiment Flow

1. The system creates simulated agents for the selected configuration.
2. Synthetic attacks and faults are injected.
3. Physical and cyber states are generated for each simulation step.
4. The anomaly layer computes deviation and anomaly probability.
5. Trend clustering and behavior updates refine context.
6. RL chooses a directional audit action.
7. Gradient descent refines the audit frequency.
8. Hybrid constraints enforce budget and capacity limits.
9. The framework compares dynamic results against a fixed baseline run.
10. Metrics are exported to logs and summaries.

## 6. Main Formulas Used In The Project

### 6.1 Live and Shared Scoring Formulas

| Formula | Meaning | Where Used |
|---|---|---|
| `z = (obs - base) / th` | normalized deviation per feature | deviation scoring |
| `dx = sqrt(mean(((x - bx)/thx)^2))` | RMS normalized physical deviation | both modules |
| `dy = sqrt(mean(((y - by)/thy)^2))` | RMS normalized cyber deviation | both modules |
| `S_i(t) = w_i (dx + dy)` | final anomaly score for agent `i` | both modules |
| `hybrid_conf = w_dev * dev_conf + w_prob * prob_support + suspicion_credit` | calibrated hybrid anomaly confidence | live scoring pipeline |
| `a_i(t) = 1` when calibrated probability and deviation conditions are jointly satisfied | anomaly flag rule | both modules |

### 6.2 Explainability Formula

| Formula | Meaning | Where Used |
|---|---|---|
| `c_j = ((x_j - b_j)/th_j)^2` | squared contribution of feature `j` | XAI |
| `relative_j = c_j / sum(c)` | percentage contribution of feature `j` | XAI |

### 6.3 Gradient Audit Optimization Formula

| Formula | Meaning | Where Used |
|---|---|---|
| `C_i = C_a f_i + C_f (R_i / f_i)` | cost for auditing agent `i` | experiment hybrid scheduler |
| `dC/df = C_a - C_f (R_i / f_i^2)` | gradient of audit cost w.r.t. frequency | gradient refinement |
| `f_i <- f_i - lr * dC/df` | one gradient descent update step | gradient refinement |

### 6.4 RL Formula

| Formula | Meaning | Where Used |
|---|---|---|
| `Q(s,a) <- Q(s,a) + alpha [r + gamma max_a' Q(s',a') - Q(s,a)]` | Bellman Q-learning update | RL audit scheduler |

### 6.5 Evaluation Formulas

| Formula | Meaning | Where Used |
|---|---|---|
| `Attack Reduction = (baseline - dynamic) / baseline` | attack-rate reduction | experiment evaluation |
| `Cost Efficiency = (baseline_cost - dynamic_cost) / baseline_cost` | cost saving vs baseline | experiment evaluation |
| `Risk Mitigation = (raw_risk - effective_risk) / raw_risk` | mitigation achieved by audit and response | experiment evaluation |
| `Accuracy = (TP + TN)/(TP + TN + FP + FN)` | overall correctness | experiment evaluation |
| `Recall = TP/(TP + FN)` | anomaly sensitivity | experiment evaluation |
| `F1 = 2 * (Precision * Recall)/(Precision + Recall)` | balance of precision and recall | experiment evaluation |

### 6.6 Rapid SCADA Calculated Channel Formula Type

The live SCADA model also uses calculated-channel expressions of the form:

`value = f(input_channels, time, drift_factor)`

This is what creates live, changing telemetry inside Rapid SCADA for the prototype.

## 7. Main Algorithms In The Project

## 7.1 Deviation-Based Scoring

This is the core explainable score used by the live SCADA path and by the merged experiment-live pipeline.

What it does:

- computes physical deviation
- computes cyber deviation
- combines them using criticality weight
- triggers audit action based on threshold

Main file:

- `smartgrid_mas/behavior_analysis/deviation_score.py`

## 7.2 LSTM Anomaly Detection

This is the sequence model used in the experiment module and also in the live experiment pipeline when the live checkpoint is available.

What it does:

- converts time series into sliding windows
- trains an LSTM anomaly detector
- predicts anomaly probability from historical windows
- calibrates probability using validation-time temperature scaling and decision threshold fitting
- uses stronger imbalance handling with oversampling, cost-sensitive loss, and focal loss
- supports supervised training and inference

Main files:

- `smartgrid_mas/anomaly_detection/lstm_model.py`
- `smartgrid_mas/anomaly_detection/train_lstm.py`
- `smartgrid_mas/anomaly_detection/inference.py`
- `smartgrid_mas/anomaly_detection/dataset.py`

## 7.3 Q-Learning Audit Scheduler

This is the RL part of the dynamic audit scheduler.

What it does:

- encodes agent state
- selects actions using epsilon-greedy policy
- updates Q-values with Bellman learning
- learns directional audit actions such as increase, decrease, or hold

Main files:

- `smartgrid_mas/audit/audit_scheduler_rl.py`
- `smartgrid_mas/audit/state_encoder.py`
- `smartgrid_mas/audit/schedule_step.py`

## 7.4 Gradient-Based Audit Refinement

This is the continuous optimization part of the scheduler.

What it does:

- refines audit frequencies after RL action selection
- uses the audit cost function gradient
- keeps frequencies inside allowed bounds

Main files:

- `smartgrid_mas/audit/gradient_update.py`
- `smartgrid_mas/audit/gradient_step.py`

## 7.5 Hybrid Scheduler

This is the combined RL + gradient + constraints pipeline.

What it does:

- uses RL for discrete action direction
- uses gradient descent for continuous refinement
- uses constraints for budget and max-audit enforcement
- allocates audit resources under operational limits

Main file:

- `smartgrid_mas/audit/hybrid_scheduler.py`

## 7.6 Trend Clustering and Behavior Adaptation

This is the adaptive behavior layer.

What it does:

- computes trend features
- clusters agent trajectories
- updates baselines and thresholds
- adjusts behavior context over time

Main files:

- `smartgrid_mas/behavior_analysis/trend_features.py`
- `smartgrid_mas/behavior_analysis/trend_clustering.py`
- `smartgrid_mas/behavior_analysis/baseline_update.py`
- `smartgrid_mas/behavior_analysis/threshold_update.py`
- `smartgrid_mas/behavior_analysis/behavior_pipeline.py`

## 7.7 Response and Severity Logic

This is the mitigation and audit-to-response part.

What it does:

- classifies severity
- applies response actions
- records mitigation events
- updates response state for downstream metrics

Main files:

- `smartgrid_mas/response/response_controller.py`
- `smartgrid_mas/response/severity_scoring.py`
- `smartgrid_mas/response/mitigation_actions.py`
- `smartgrid_mas/response/impact_factor.py`

## 7.8 XAI / Explanation Layer

This is the explainability module.

What it does:

- ranks physical and cyber feature contributions
- explains audit decisions
- exports reason traces

Main files:

- `smartgrid_mas/xai/explain.py`
- `smartgrid_mas/xai/export_shap_reasons.py`

## 7.9 Federated and Blockchain Components

These are support modules for extension and logging.

What they do:

- federated aggregation support
- blockchain-style audit anchoring and event integrity logging

Main files:

- `smartgrid_mas/federated/fedavg.py`
- `smartgrid_mas/federated/orchestrator.py`

## 8. What Changed From The Base Paper

## 8.1 What The Base Paper Gives

The base paper contributes:

- anomaly detection + audit coupling
- dynamic audit optimization idea
- strong conceptual multi-agent framing
- simulated benchmark results

## 8.2 What This Project Adds

This project extends the base paper into an actual software platform.

Major additions:

1. **Live SCADA integration**
   - Rapid SCADA Webstation used as live telemetry source

2. **Experiment / live separation**
   - dashboard split into two clean workspaces

3. **Batched SCADA ingestion**
   - fixed the per-agent rate-limit problem

4. **Explainable live path**
   - live decisions are based on deviation logic plus feature-level explanation

5. **100-agent live validation**
   - the live system is validated at full 100-agent scale

6. **Operator-facing dashboard**
   - audit trail, risk analytics, response workflow, XAI, topology, system health

7. **Cyber-channel enrichment**
   - base `300` channels extended with `370` addon channels for packet loss, integrity, communication frequency, and additional latency coverage

8. **Merged live experiment pipeline**
   - live SCADA batches now pass through deviation scoring, optional live LSTM inference, adaptation, hybrid scheduling, response, and XAI

9. **Fresh-source dashboard mapping**
   - live SCADA pages map only current `/grid/status` and `/v1/scada/live` state
   - latest experiment cards map from `/v1/runs/latest` instead of stale shared summaries

10. **Implementation realism**
   - complete backend, bridge, dashboard, and SCADA integration

## 8.3 Why This Project Performs Better

The main reasons the proposed project improves over the conceptual base paper are:

- stronger operational integration
- better audit-resource targeting
- explicit criticality-weighted anomaly scoring
- explainable audit decisions rather than opaque outputs
- batched SCADA telemetry ingestion instead of unstable per-agent polling
- improved operator visibility through dashboards and audit trail

## 8.4 Algorithm-Level Difference vs Base Paper

| Area | Base Paper | Proposed Project |
|---|---|---|
| anomaly-to-audit coupling | conceptual | implemented end-to-end |
| SCADA integration | absent | Rapid SCADA live path |
| experiment/live separation | absent | explicit dashboard workspace split |
| live explainability | limited | feature-level XAI and decision explanation |
| audit execution path | conceptual | backend + response + dashboard trail |
| probability calibration | not explicit | calibrated LSTM probability + hybrid scoring gate |
| imbalance handling | not explicit | oversampling + cost-sensitive loss + focal-loss training |
| deployment validation | limited | 100-agent live verification |

## 15. Quick “What Makes This Project Good”

The project is strong because it does not stop at anomaly detection.

It combines:

- anomaly scoring
- audit decision logic
- explainability
- SCADA integration
- experiment benchmarking
- live operational views

The biggest improvement over the base paper is not just a different metric value.
It is that this project turns the anomaly-and-audit idea into a complete operational pipeline that can be demonstrated, validated, and defended.
