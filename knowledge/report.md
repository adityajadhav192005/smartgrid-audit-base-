## 1 Introduction

### 1.1 Background

Modern smart grids are cyber-physical systems in which electrical assets, communication networks, control software, and operator workflows interact continuously. This improves efficiency, renewable integration, and observability, but it also expands the attack surface and increases operational complexity. Recent reviews show that anomaly detection in smart grids is no longer a narrow electrical-engineering problem; it is now a joint problem of cybersecurity, data analytics, distributed control, and resilience engineering [2]-[5].

Multi-agent system thinking is particularly relevant in this setting. Generators, substations, PMUs, breakers, and controllers can be modeled as semi-autonomous entities that observe local conditions and contribute to system-wide decisions. However, as Farid noted, many multi-agent power-system implementations improve decentralization without fully addressing resilience and adversarial behavior [6]. This gap is important because smart-grid communication paths, SCADA layers, and monitoring devices remain vulnerable to false data injection, replay, delay, and coordinated cyber-physical disruption [4], [7]-[11].

The project developed here sits at the intersection of:

- smart-grid anomaly detection
- risk-aware audit scheduling
- SCADA-integrated cyber-physical monitoring
- explainable operational decision support

In implementation terms, the framework combines:

- an experiment workspace driven by latest-run artifacts
- a live Rapid SCADA workspace driven by actual Rapid SCADA current data

This is important because the final system is not only a simulation runner. It is also a working SCADA-to-AI audit pipeline.

### 1.2 Motivation

Static auditing and static thresholds are weak fits for modern smart grids. The main operational tension is that over-auditing wastes resources, while under-auditing creates blind spots during evolving attacks. Literature on false data injection, PMU data integrity, and AMI attack detection consistently shows that adversaries can exploit both cyber and physical layers if monitoring is rigid or fragmented [8]-[12].

The project was motivated by three practical costs:

1. audit and monitoring cost
2. missed-attack or delayed-response cost
3. false-positive cost through alert fatigue and unnecessary intervention

The goal, therefore, is not just to detect anomalies but to allocate audit attention more intelligently while keeping the system explainable and operationally stable.

### 1.3 Aim

To design and validate an explainable SmartGrid AI audit framework that integrates cyber-physical anomaly detection, adaptive audit decisioning, and live Rapid SCADA telemetry while preserving operational clarity and reproducibility.

### 1.4 Research Objectives

1. Build a hybrid cyber-physical anomaly analysis layer using interpretable deviation scoring.
2. Implement adaptive audit decision logic that prioritizes higher-risk assets under cost constraints.
3. Separate experiment telemetry from live SCADA telemetry so the dashboard remains technically correct.
4. Integrate Rapid SCADA as a live supervisory source for a `10 x 10` smart-grid asset model.
5. Evaluate the framework using actual run logs across multiple grid sizes and summarize its strengths and limitations honestly.

### 1.5 Scope

The project now supports:

- experiment runs for `N=100`, `N=200`, and `N=500`
- a live Rapid SCADA workspace with `100` live agents

The live SCADA pipeline is real, but the current SCADA telemetry is generated by calculated channels inside Rapid SCADA rather than coming from physical field devices. This distinction is important for academic honesty and is preserved throughout this report.

## 2 Literature Survey

### 2.1 Related Research Themes

The relevant literature can be grouped into five connected themes.

#### A. Smart-grid anomaly detection and review literature

Recent reviews confirm that anomaly detection in smart grids is shifting toward hybrid, AI-supported, and framework-based approaches rather than isolated threshold rules [2], [3]. These reviews also show that many studies focus on only one layer of the problem, such as cyber attack detection or energy anomaly detection, without integrating audit policy or operator response [2], [3].

#### B. Smart-grid cybersecurity architecture and communication security

Smart-grid cybersecurity depends heavily on communication design, protocol trust, and architectural separation. Surveys by Achaal et al. and Sharma show that attack surfaces span HAN, NAN, WAN, AMI, SCADA, PMUs, and edge-control layers, and that secure communication is central to resilient CPS operation [4], [5]. Sadeghi et al. similarly emphasize that industrial systems need layered defenses rather than single-point controls [7].

#### C. Multi-agent resilience in power systems

Farid's work on resilient multi-agent design for future power systems is relevant because it frames resilience as an architectural property rather than merely a local algorithmic one [6]. This supports the design choice in this project to treat the grid as an interacting agent ecosystem rather than a flat telemetry table.

#### D. False data injection, PMU integrity, and SCADA-aware attack detection

The false-data-injection literature is foundational. Liu, Ning, and Reiter showed that attackers can alter state estimation while evading conventional bad-data detection [8]. Giani et al. demonstrated the importance of PMU placement in recognizing unobservable integrity attacks [9]. Later work extended this line by analyzing coordinated attackers [10], expectation-maximization-based detection [11], and PMU-based anomaly and replay detection [15]. These studies justify the emphasis in this project on combining electrical and cyber indicators instead of trusting a single telemetry stream.

#### E. Intrusion detection and reinforcement learning for CPS security

Korba et al. presented anomaly-based detection for AMI overloading attacks [12], while Martinelli et al. used SCADA-derived logs for smart-grid intrusion detection [13]. Ibrahim and Elhafiz explored reinforcement learning for CPS security analysis [14]. Together, these works suggest that operational protection needs both detection and adaptation, but many prior systems still stop at classification and do not continue into audit allocation and dashboard-visible control logic.

The literature review indicates four practical gaps:

1. many papers detect anomalies but do not optimize or explain audit decisions
2. many surveys discuss architecture, but not end-to-end working implementations
3. many smart-grid security studies remain offline or dataset-only, without a live SCADA path

## 3 Proposed System Architecture Design

### 3.1 Final System Architecture

The final architecture has two operational workspaces:

- `Experiment Running`
- `Rapid SCADA Live`

This split is deliberate. Experiment pages always reflect the latest completed run. Rapid SCADA pages always reflect live SCADA state. This avoids one of the earlier architectural issues, where experiment summaries and live SCADA views could conflict at the UI level.

At the control-stack level, the project uses:

- Rapid SCADA Webstation on `http://127.0.0.1:10109`
- PowerShell bridge in `scripts/pull_rapidscada_to_api.ps1`
- FastAPI backend in `smartgrid_mas/api/app.py`
- Next.js dashboard in `web`

The live SCADA workspace is modeled as a fixed `10 x 10` asset grid:

- `GEN-01` to `GEN-20`
- `SUB-21` to `SUB-50`
- `PMU-51` to `PMU-75`
- `BRK-76` to `BRK-100`

### 3.2 End-to-End SCADA Data Flow

The live SCADA pipeline is:

1. Rapid SCADA calculated channels generate current values
2. the bridge authenticates to Rapid SCADA Web API
3. the bridge reads all required channels
4. the bridge posts one batched 100-agent payload to `/v1/scada/ingest/tags/batch`
5. the backend normalizes the tags into physical and cyber vectors
6. the backend computes score, severity, audit action, and explanation
7. the dashboard renders live SCADA state

Earlier versions used one request per agent and triggered `429 Too Many Requests`. Batch ingest fixed that instability.

### 3.3 Detection and Audit Logic

The live SCADA path uses an explainable deviation model:

$$
S_i(t)=w_i(d_x+d_y)
$$

where:

- `d_x` is physical deviation from baseline
- `d_y` is cyber deviation from baseline
- `w_i` is the agent criticality weight

The decision logic is intentionally simple:

- if score exceeds the threshold, increase audit
- otherwise maintain audit

This is weaker than a fully adaptive learning policy, but it is easier to explain and more defensible in a viva setting.

For the experiment-mode pipeline, the system uses a three-layer multi-detector architecture that extends the single-threshold detector above. Layer A is the calibrated LSTM threshold (probability >= 0.80, score >= 3.60 in the ROBUST profile). Layer B is a temporal accumulator that flags any agent whose LSTM probability remains at or above 0.55 for five or more consecutive timesteps - this catches sustained low-amplitude attacks (FDI, MITM) that never spike high enough for Layer A. Layer C is a set of attack-type-specific sub-detectors:

- C-1, a two-sided CUSUM drift test on physical residuals scaled by sensor thresholds, detects FDI bias attacks
- C-2, a 2-of-3 explicit network rule on latency multiplier (>=3x), packet-loss floor (>=0.15), and comm-frequency drop (>=0.40), detects DoS attacks
- C-3, a combined integrity-drop and temporal z-jump check, detects MITM tampering

Layer outputs are combined with OR-with-precedence: any layer firing flags the agent, but type-specific labels (FDI/DOS/MITM) take precedence over the generic SUSTAINED label produced by Layer B. The combiner prevents the false-positive inflation that naive ensemble averaging would cause. Implementation in `smartgrid_mas/detection/multilayer_detection.py`; wiring in `smartgrid_mas/behavior_analysis/scoring_pipeline.py::compute_score_and_flag`.

### 3.4 Expected Normal Values

The live SCADA algorithm compares observed values against expected normal values. These are type-specific baselines, for example:

- generators:
  - voltage `230`
  - frequency `50`
  - current `15`
- substations:
  - load-centered profile with cyber latency monitoring
- PMUs:
  - voltage around `230`
  - frequency around `50`
- breakers:
  - status-centered monitoring with low current/power expectation

These expected values are hand-set engineering baselines, not adaptive learned profiles.

### 4.2 Rapid SCADA Engineering Outcome

The Rapid SCADA project now contains a `300`-channel model for the `100`-agent grid. A critical implementation lesson was discovered during integration:

- formulas alone were not enough
- the channels had to be imported as `Calculated` channels, not plain `Input` channels

Once `CnlTypeID=3` was used in the generated `Cnl.xml`, Rapid SCADA began producing current data for all `100` agents.

### 4.3 What Is Real Versus Simulated

This project now has a real live SCADA transport path, but the current values are still generated by calculated channels inside Rapid SCADA. Therefore:

- live SCADA runtime: yes
- live current data retrieval: yes
- direct physical field telemetry: no

This is still a valid cyber-physical prototype because the integration, scoring, dashboard, and audit path are all operational.

### 4.4 Base Paper Alignment

The current implementation is directly related to the base paper by Priyadarsini [1], but it extends it in more operational directions:

- clearer experiment-versus-live separation
- actual SCADA integration and verification
- batch ingest and API hardening
- richer operator dashboard structure
- explicit provenance and fallback behavior

In other words, the project is not only a conceptual replication. It is an engineering extension of the base idea.

### 6.3 Interpreting Accuracy Versus F1

The logged runs show very high accuracy but much lower F1. This is not a contradiction. The summaries explicitly note:

- `validity_notes: extreme_attack_density (>50%)`

and the class distributions are highly imbalanced. For example, the `NONE` and `FAULT` categories dominate event volume in several runs, so raw accuracy remains high even when minority attack detection is harder. This is why the F1 score is the better signal of classification difficulty in these scenarios.

- `N=200` p95 end-to-end delay: `92.29 ms`
- `N=500` p95 end-to-end delay: `232.96 ms`

This shows that the system remains usable at higher sizes, but responsiveness degrades noticeably as scheduling and action logic process more agents.

### 6.5 SCADA Deployment Result

The Rapid SCADA workspace now achieves the strongest deployment result in the project:

- `100/100` live agents in the SCADA grid

This matters because the project now demonstrates both:

- experiment-scale evaluation through the run logs
- working live SCADA integration through the 100-agent Rapid SCADA grid

### 6.6 Overall Findings

The most defensible findings from the real logs are:

1. the hybrid approach is substantially more cost-efficient than the fixed baseline
2. the system preserves very high raw accuracy across tested scales
3. the tuned `N=100` thesis profile surpasses the base-paper benchmark on risk mitigation while retaining high cost efficiency
4. risk mitigation remains positive, but weakens as topology size grows
5. end-to-end delay remains manageable, but increases materially at `N=500`
6. the live SCADA path is operational and no longer limited to a few channels

## 7 Discussion

### 7.1 What the System Does Well

The project performs well in four areas.

#### A. End-to-end integration

It is no longer only an experimental framework. It now supports:

- latest-run experiment analytics
- live Rapid SCADA telemetry
- anomaly scoring
- audit decisions
- explainability
- operator dashboards

#### B. Operational clarity

The workspace split between experiment and live SCADA is a real architectural improvement. It reduces confusion and makes the system easier to explain and defend.

#### C. Cost-aware control

The logs show strong cost-efficiency gains, especially at larger scales.

#### D. Explainability

The live algorithm is simple enough to justify in a viva and transparent enough to support operator trust.

### 7.2 Current Limitations

The current limitations should be stated honestly.

1. the live SCADA algorithm is still baseline and threshold driven
2. expected normal values are hand-set rather than learned adaptively
3. the live SCADA telemetry is simulated/calculated, not physical field data
4. class imbalance and extreme attack-density conditions reduce F1 even when raw accuracy stays high
5. larger topologies increase delay and weaken risk-mitigation performance

### 7.3 Why This Is Still a Strong Project

Even with those limitations, the project is strong because it demonstrates:

- a real live SCADA-to-AI path
- a working 100-agent supervisory model
- actual run-backed performance metrics
- explainable audit logic
- an operator-oriented dashboard architecture

That is more defensible than claiming a fully autonomous smart-grid defender without deployment realism.

## 8 Conclusion and Future Work

### 8.1 Conclusion

This project delivers a working SmartGrid AI audit framework that combines:

- experiment-based evaluation
- live Rapid SCADA integration
- explainable cyber-physical anomaly scoring
- audit decision support
- structured operator dashboards

The final implementation goes beyond the base paper by operationalizing the architecture. It now supports a verified `100/100` live Rapid SCADA grid, batch SCADA ingest, separated workspaces, and logged experiment results across multiple scales.

The strongest evidence from the logs is that the framework consistently improves cost efficiency while maintaining high accuracy, although F1 and risk-mitigation performance degrade with larger topologies and class imbalance.

### 8.2 Future Work

The most meaningful next steps are:

1. adaptive baselines per asset rather than fixed normal values
2. temporal anomaly detection rather than pointwise deviation only
3. context-aware thresholds for peak load, off-peak, maintenance, and fault states
4. integration with real industrial devices or protocol-level field data
5. stronger minority-attack handling to improve F1 under extreme class imbalance

The correct final academic position is:

This is a strong explainable cyber-physical operational prototype with real live SCADA integration, but it is not yet a physical utility deployment.
