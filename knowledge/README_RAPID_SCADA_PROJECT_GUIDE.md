## 1. What Rapid SCADA Is And Why We Use It

### 1.1 What Is Rapid SCADA

Rapid SCADA is a free, open-source Russian SCADA (Supervisory Control and Data Acquisition) system. It provides:

- A real-time tag/channel database for storing and computing process values
- A web-based operator interface (Webstation)
- A REST HTTP API (`GetCurData`) for reading current channel values programmatically
- Support for calculated channels — channels whose values are computed by internal formulas rather than coming from physical hardware

**Why we chose Rapid SCADA over alternatives:**

| Option | Why Rejected / Why Chosen |
|--------|--------------------------|
| OpenSCADA | Less documentation, harder Windows setup |
| WinCC (Siemens) | Commercial, expensive, requires industrial hardware |
| Ignition SCADA | Commercial, expensive |
| Pure simulation (CSV/Python) | No real SCADA pipeline, weaker demo |
| **Rapid SCADA** | ✅ Free, open-source, real HTTP API, installable on Windows, realistic OT demo layer |

Rapid SCADA gives us a **real SCADA pipeline** — authentication, API, channel model, Webstation UI — even though the current values are generated by calculated channels rather than physical field devices.

### 1.2 What "Calculated Channels" Means

In Rapid SCADA, channels can be:

- **Input channels** — receive data from hardware (PLCs, RTUs, Modbus, OPC UA)
- **Calculated channels** (`CnlTypeID = 3`) — compute values from formulas inside SCADA

Our 100-agent grid uses calculated channels. This means:

- **The SCADA pipeline is real** (authentication, API, bridge, backend, dashboard all work)
- **The current values are simulated inside SCADA** (formulas produce voltage, current, load, frequency, etc.)
- **This is standard practice** for academic testbeds — even ORNL's PowerCyber testbed uses simulated process values

**Critical engineering lesson discovered during integration:**

> Channels with formulas but wrong type (`CnlTypeID = 1` Input) do NOT publish current data.
> Only after switching to `CnlTypeID = 3` (Calculated) did all 100 agents become live.

## 2. The 100-Agent Grid Layout

### 2.1 Agent Types and Channel Mapping

The live grid uses a fixed 100-agent model:

| Agent Range | Type | Count | Physical Tags | Cyber Tags |
|-------------|------|-------|--------------|-----------|
| GEN-01 to GEN-20 | Generator | 20 | voltage, current | latency, packet_loss, integrity, comm_freq |
| SUB-21 to SUB-50 | Substation | 30 | substation_load, latency | packet_loss, integrity, comm_freq |
| PMU-51 to PMU-75 | PMU | 25 | voltage, frequency | latency, packet_loss, integrity, comm_freq |
| BRK-76 to BRK-100 | Breaker | 25 | breaker_status | latency, packet_loss, integrity, comm_freq |

**Total channels:** 300 base + 370 cyber addon = 670 configured channels

### 2.2 What Each Tag Means

**Physical tags (from real SCADA channels):**
- `voltage` — line voltage in volts (nominal 230 V)
- `current` — line current in amps
- `frequency` — grid frequency in Hz (nominal 50 Hz)
- `substation_load` — feeder load in kW
- `breaker_status` — 1.0 = closed, 0.0 = open

**Cyber tags (synthetic baseline, not from real IDS/SIEM):**
- `latency` — communication delay in ms (baseline: GEN 3ms, SUB 4ms, PMU 2ms, BRK 3ms)
- `packet_loss` — fraction of dropped packets (baseline: 0.001 = 0.1%)
- `integrity` — message integrity score (baseline: 1.0 = perfect)
- `comm_freq` — communication frequency in Hz (baseline: 50.0)

**Why cyber tags are synthetic — honest explanation for the viva:**

> Rapid SCADA is a process-control system, not a network monitor. Packet loss, integrity, and comm_freq come from IDS/SIEM tools like Snort or Suricata. For this research prototype, we use engineered baselines — the same approach used by the base paper and every comparable academic CPS testbed. In production, these would come from a network monitoring agent. This is documented as future work.

### 2.3 Derived Tags

The SCADA adapter derives missing tags automatically:

| Derived Tag | Formula | When Applied |
|-------------|---------|-------------|
| `power` | `(voltage × current × 0.92) / 1000` | when power not present but V and I are |
| `latency` | copied from `response_time` | when latency absent but response_time present |
| `breaker_status` | `1.0 if current > 0.5 else 0.0` | when breaker_status absent |

```

         │

         ├── ingest_snapshot()            → store per-agent state
         │
Next.js Dashboard (http://127.0.0.1:3000)
         │

         ├── /rapid-scada/risk            → risk analytics
         ├── /rapid-scada/monitor         → live monitoring
         ├── /rapid-scada/threats         → threat events
         ├── /rapid-scada/audits          → audit trail

         ├── /rapid-scada/assets          → topology view
         ├── /rapid-scada/grid            → SCADA grid iframe
         └── /rapid-scada/connectivity    → connection health
```

### 3.1 Why Batch Ingest (Not Per-Agent)

Earlier design posted one HTTP request per agent (100 requests per poll cycle). This caused:
- HTTP 429 Too Many Requests errors
- Race conditions in state updates
- Unstable live telemetry

**Fix:** The bridge batches all 100 agents into a single `/v1/scada/ingest/tags/batch` request. One poll = one request. This stabilized the entire live path.

## 4. The Bridge Script

**File:** `scripts/pull_rapidscada_to_api.ps1` (830 lines PowerShell)

### 4.1 What It Does

1. Reads authentication credentials for Rapid SCADA
2. For each of 100 agents, reads assigned SCADA channel numbers
3. Calls `GetCurData` API with those channel IDs
4. Assembles a tag dictionary per agent
5. Optionally injects attack signatures (GeneratorFDI, SubstationDoS, PMUDesync, BreakerTrip)
6. Posts the full 100-agent batch to the backend
7. Loops every N seconds (configurable)

### 4.2 Attack Simulation Modes

The bridge supports 5 demo modes:

| Mode | What It Simulates |
|------|------------------|
| `Normal` | All agents at nominal values |
| `GeneratorFDI` | False data injection on GEN agents (voltage spike, packet_loss ↑, integrity ↓) |
| `SubstationDoS` | DoS attack on SUB agents (high latency, low comm_freq) |
| `PMUDesync` | PMU phase desynchronization (frequency deviation, integrity ↓) |
| `BreakerTrip` | Spurious breaker trip (status → 0, current → 0) |
| `Independent` | Random independent anomalies at configured rate |
| `Auto` | Rotates through attack types automatically |

### 4.3 Agent Profile Defaults (What The Bridge Sends)

```
GEN: voltage=230V, current=15A, latency=3ms, packet_loss=0.001, integrity=1.0, comm_freq=50Hz
SUB: load=180kW, latency=4ms, packet_loss=0.001, integrity=1.0, comm_freq=50Hz
PMU: voltage=230V, frequency=50Hz, latency=2ms, packet_loss=0.001, integrity=1.0, comm_freq=50Hz
BRK: status=1.0, latency=3ms, packet_loss=0.001, integrity=1.0, comm_freq=50Hz
```

## 5. The SCADA Adapter

**File:** `smartgrid_mas/integration/scada_adapter.py` (347 lines)

### 5.1 Per-Agent-Type Profiles

Each agent type has its own baseline and threshold profile with ~80 environment variable overrides:

```python
GEN profile: voltage_baseline=230V, frequency_baseline=50Hz, current_baseline=15A
             latency_baseline=3ms, packet_loss_baseline=0.001, integrity_baseline=1.0
             voltage_threshold=12V, current_threshold=8A, latency_threshold=1.5ms

SUB profile: voltage_baseline=230V, current_baseline=12A, power_baseline=180kW
             latency_baseline=4ms, ...

PMU profile: voltage_baseline=230V, frequency_baseline=50Hz
             latency_baseline=2ms, ...

BRK profile: status-centered, low power/current assumptions
             latency_baseline=3ms, ...
```

### 5.2 Required Live Tags Enforcement

The adapter enforces strict validation — if required tags are missing, ingest is rejected:

```
GEN requires: voltage, current, latency, packet_loss, integrity, comm_freq
SUB requires: substation_load, latency, packet_loss, integrity, comm_freq
PMU requires: voltage, frequency, latency, packet_loss, integrity, comm_freq
BRK requires: breaker_status, latency, packet_loss, integrity, comm_freq
```

### 5.3 Engineering Scale Normalization

Raw SCADA values in [0.0, 2.0] range are scaled up to engineering units:

```python
if 0.0 <= raw_val <= 2.0:
    merged_phys[k] = raw_val * ENG_SCALE[k]
# ENG_SCALE: voltage=230.0, frequency=50.0, current=100.0, power=250.0
```

This handles the case where Rapid SCADA formulas produce normalized [0,1] outputs instead of raw engineering values.

## 6. Live Pipeline Processing

**File:** `smartgrid_mas/integration/live_experiment_pipeline.py`

After SCADA tags are received, the live pipeline runs:

1. **Deviation scoring** — compute `dx` (physical) and `dy` (cyber) RMS deviations
2. **LSTM inference** — if checkpoint available, predict anomaly probability on last 24 windows
3. **3-modality voting** — combine deviation + LSTM + behavioral signature
4. **Tier-A FP suppression** — suppress if `score_ratio < 3.5 AND no_signature AND weak LSTM`
5. **Hybrid audit scheduler** — Q-learning action + gradient refinement + budget constraints
6. **Response logic** — severity classification → mitigation action → response state
7. **XAI** — rank feature contributions, generate explanation text
8. **Snapshot storage** — store per-agent state for dashboard consumption

## 10. Viva Defense Points For SCADA

**Q: Why Rapid SCADA and not a real substation?**
A: Rapid SCADA provides a real OT integration layer with authentication, API, channel model, and Webstation — exactly the same software stack used in real deployments. Our contribution is the AI audit algorithm, not the sensor hardware. The base paper itself uses pure simulation with no SCADA at all.

**Q: Are the readings real?**
A: The SCADA pipeline is real — real API calls, real authentication, real channel model. The values are generated by calculated channels inside SCADA. This is standard for academic CPS testbeds. In production, you would replace calculated channels with channels reading from actual PLCs via Modbus/OPC UA.

**Q: What about packet_loss and integrity — are those real?**
A: No. Rapid SCADA is a process-control system, not a network monitor. Cyber metrics come from IDS tools like Snort. We use engineered baselines (packet_loss=0.001, integrity=1.0) which is the same approach as the base paper. IDS integration is documented future work.

**Q: What is the data rate?**
A: The bridge polls Rapid SCADA every 5 seconds. Each poll reads all 100 agents and posts one batch request. 12 polls per minute, 720 per hour.
